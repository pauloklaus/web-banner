# Code guide

Minimal instructions for human contributors and AI agents. Read before writing or reviewing code.

## Project language

**English is the default project language.**

| What                                                           | Language |
| -------------------------------------------------------------- | -------- |
| Documentation (`README.md`, `doc/`, `AGENTS.md`)               | English  |
| Commit messages                                                | English  |
| Issues and pull requests (title, description, review comments) | English  |
| Code identifiers (variables, functions, types, files)          | English  |

The app UI is multilingual via i18n (`en`, `pt-BR`, `es`) — that applies to **end users**, not to repo docs or git history.

## Stack

- Vue 3 (`<script setup lang="ts">`), TypeScript
- Vite, `vue-i18n`, PWA (`vite-plugin-pwa`)
- Styles: scoped CSS in components; `src/assets/base.css` for global reset only
- Dev/build via Docker: `Makefile` at the repo root

## Principles

1. **No code comments** — clear names and small functions replace explanations.
2. **Declarative naming** — `buildConfigUrl`, `toPlainText`, `FONT_HEIGHT_RATIO`; verbs for functions, nouns for constants.
3. **One public export per file** — each module exposes a single function, constant, or main component. Keep helpers private in the same file without `export`.
4. **Barrels** — folders with multiple modules expose `index.ts` that re-exports the public API. Import from `@/composables` (or the barrel path), not internal files, except within the same folder.
5. **i18n required** — all user-visible text goes through `t('key')` in all three locales: `en`, `pt-BR`, `es`.
6. **Minimal scope** — change only what is needed; do not mix in unrequested refactors.
7. **Centralized env** — `VITE_*` variables are read only in `src/config/readEnv.ts`; public constants live in `src/constants/` and are re-exported by `src/config/index.ts` (env-backed) or other barrels as needed.

## `src/` layout

```
src/
├── App.vue              # Orchestrates edit/play modes and global state
├── main.ts              # Bootstrap Vue + i18n
├── assets/              # Global CSS, action icons (SVG)
├── components/          # One SFC per component + index.ts (barrel)
├── composables/         # use*.ts + index.ts (barrel)
├── config/              # readEnv.ts + index.ts (barrel; env-backed constants)
├── constants/           # Shared constants + index.ts (barrel)
├── i18n/                # setupI18n.ts, locales/ + index.ts (barrel)
├── types/               # Shared types + index.ts (barrel)
└── utils/               # Pure functions + index.ts (barrel)
```

## Vue

- Always `<script setup lang="ts">`; no Options API.
- Components in `PascalCase.vue`; one component per file.
- Business logic and side effects in composables, not large SFCs.
- `defineModel` for two-way props (`v-model:message`, etc.).
- **Scoped** styles with BEM: `.edit-panel__field`, `.play-banner__text`.
- `aria-label` and button labels via i18n.
- Action icons: `src/assets/icons/` through `ActionIcon.vue`.

## Composables

- File `useName.ts` exports `function useName()`.
- One composable per responsibility (`useScrollBanner`, `useUrlSync`, `usePwaInstall`).
- Return only what the consumer needs.

## i18n

- Nested camelCase keys (`edit.message`, `play.stop`, `meta.description`).
- When adding a key, update **all three** JSON files in `src/i18n/locales/`.
- Per-locale OG metadata: `meta.ogImage`, `meta.ogLocale`.

## Config and environment

| Variable               | Purpose                         |
| ---------------------- | ------------------------------- |
| `VITE_APP_NAME`        | Name in UI, title, PWA manifest |
| `VITE_AUTHOR`          | Footer                          |
| `VITE_GITHUB_REPO_URL` | Repository link in footer       |
| `VITE_SITE_URL`        | Canonical URL (OG, meta)        |

Copy `.env.example` to `.env` for local development.

## Data and URL

- Message limited to 100 characters; sanitize with `toPlainText`.
- Shareable state via query params (`m`, `b`, `f`, `s`, `p`) — see `utils/encodeQueryConfig.ts`, `utils/applyQueryConfig.ts`, and `composables/useUrlSync.ts`.
- Colors validated/sanitized before persisting in the URL.

## Icons

| Kind                                      | Location                                          |
| ----------------------------------------- | ------------------------------------------------- |
| App icon, favicon, OG                     | `public/icons/` — generated by `make icons`       |
| Button icons (play, stop, share, install) | `src/assets/icons/` — SVGs imported in components |

Do not edit PNGs in `public/icons/` by hand; change `public/icons/icon.svg` and run `make icons`.

## Imports

- Alias `@` points to `src/` (configured in `vite.config.ts`).
- Import from barrels: `@/composables`, `@/components`, `@/config`, `@/constants`, `@/utils`, `@/types`, `@/i18n`.
- Direct file imports within the same folder are allowed (e.g. `./ActionIcon.vue`).
- Local TS module imports without file extensions.
- Suggested order: npm packages → barrels → assets.

## Avoid

- Comments (`//`, `/* */`, `<!-- -->` in code)
- Hardcoded UI strings outside i18n
- `import.meta.env` outside `src/config/readEnv.ts`
- Secrets in the repository (`.env` is not committed)
- Extra UI libraries, global CSS in components
- Unrelated changes in the same delivery

## Code review

When reviewing a diff or PR, check:

- [ ] No new comments
- [ ] UI text in all 3 locales
- [ ] Self-explanatory names
- [ ] New logic in composables/utils when appropriate
- [ ] Minimal scope
- [ ] `npm run typecheck` passes
- [ ] `make test` passes
- [ ] `make coverage` passes (≥80% global coverage)
- [ ] `make build` passes

## Tests

Unit tests use Vitest (`make test`). Coverage is enforced by `make coverage` with a **80% minimum** on lines, branches, functions, and statements (see `vite.config.ts`). Excluded from coverage: `src/types/`, `src/constants/`, barrels (`index.ts`), bootstrap (`main.ts`), test helpers (`src/test/`), and type shims.

## Commits

Use [Conventional Commits](https://www.conventionalcommits.org/) in **English**.

### Format

```
type(scope): short imperative summary
```

- **type** — what changed (see table below)
- **scope** — optional; area of the codebase (`play`, `i18n`, `pwa`, `utils`, …)
- **summary** — lowercase, imperative mood, no trailing period; explain _why_ when useful in the body

### Types

| Type       | When to use                                          |
| ---------- | ---------------------------------------------------- |
| `feat`     | New feature or user-visible behavior                 |
| `fix`      | Bug fix                                              |
| `docs`     | Documentation only                                   |
| `refactor` | Code change without fixing a bug or adding a feature |
| `perf`     | Performance improvement                              |
| `test`     | Tests                                                |
| `chore`    | Tooling, dependencies, housekeeping                  |
| `ci`       | CI/CD configuration                                  |
| `style`    | Formatting, whitespace (no logic change)             |

### Examples

```
feat(share): add WhatsApp fallback on mobile
fix(play): prevent descenders from being clipped
docs: add typecheck target to Makefile
refactor(utils): split query config into single-export modules
chore(deps): bump vue-tsc
```

### Rules

- One logical change per commit when possible
- Do not mix unrelated changes in the same commit
- Prefer `feat` / `fix` / `docs` over vague messages like `update` or `wip`

## Releases

Cloudflare Pages deploys automatically on every push to `main`. Keep **one deploy per merge** by bumping the version in the pull request — do not rely on post-merge commits.

| PR title or branch prefix | Version bump |
| ------------------------- | ------------ |
| `fix:` or `fix/`          | patch        |
| `feat:` or `feat/`        | minor        |
| `breaking:` or `breaking/` | major       |
| `docs:`, `chore:`, `ci:`, … | none      |

Before opening a release PR, run the matching command locally:

```bash
npm version patch --no-git-tag-version   # fix
npm version minor --no-git-tag-version   # feat
npm version major --no-git-tag-version   # breaking
```

CI checks that `package.json` matches the expected version. After merge, the Release workflow tags `vX.Y.Z` from `package.json` without pushing another commit to `main`.

Use **squash merge** so the commit message on `main` matches the PR title.
